# Огляд агентних інструментів ragnarbot — рішення-орієнтований звіт

> Контекст: ragnarbot — легкий (~4 700 LOC) async-Python *персональний асистент* у Telegram (не IDE). Працює з файлами у воркспейсі користувача, виконує shell, ходить у веб, тримає пам'ять/нотатки, планує cron, запускає фонові процеси та суб-агентів. LLM через LiteLLM. Звіт синтезує майстер-список ідей, оцінки трьох лінз (product-value / cost-feasibility / snake-oil-skeptic), вихід gap-критика та технічний аудит.

---

## 1. Короткий підсумок

- **Пошуку немає взагалі.** У жодному реєстрі (`loop.py:245-305` і ізольований `2681+`) немає інструмента пошуку по вмісту чи glob. «Знайди, де я згадував X у нотатках» — найчастіша операція асистента — йде через `exec+grep`, який ламається на подвійному екрануванні (`shell.py:73`), мовчки усікає на 10 000 символах (`shell.py:116-118`) і хибно блокується safety-guard на будь-якому `/` чи `../` (`shell.py:141,147`). **Найбільший виграш — додати `grep` і `glob` першого класу.**
- **`file_read` небезпечніший за `cat`.** Читає весь файл без offset/limit/cap і без номерів рядків (`filesystem.py:71-72`). Файл 1 МБ ≈ 250-300K токенів → hard-fail «prompt too long» або тихе усічення всієї розмови; результат переб'ється на кожній з ~20 ітерацій циклу.
- **Пам'ять майже недосяжна.** `get_memory_context()` віддає лише MEMORY.md + сьогодні/вчора (`memory.py:109-132`); нотатки тижневої давнини невидимі, а MEMORY.md роздуває кожен промпт. **`search_memory` на SQLite FTS5 (без нової залежності) — головний продуктовий виграш.**
- **Немає центрального бюджету виводу.** `ToolRegistry.execute` і `add_tool_result` додають результат «як є» (`registry.py:60`, `context.py:440-446`); кепи розкидані й непослідовні (exec 10K, web_fetch 50K, file_read — 0). Один великий результат переб'ється кожну ітерацію.
- **`edit_file` має баг тихого успіху.** При неоднозначному збігу повертає `Warning:` як успішний результат і нічого не змінює (`filesystem.py:198-200`); плюс строгий exact-match без толерантності до пробілів/CRLF — це #1 реальна причина відмов правок.
- **`exec` втрачає діагностику при усіченні.** stderr+код виходу додаються в кінець, а усічення — head-only (`shell.py:100-118`) → агент вважає, що зламана команда успішна; timeout повертає порожнечу.
- **Дві справжні діри безпеки.** (1) Реального `max_iterations` немає — у коді `while True`, хоч CLAUDE.md обіцяє 20 → необмежені витрати, особливо в cron/hook/heartbeat без користувача. (2) Неконтрольовані великі читання.
- **Цілі зони не охоплені ідеацією** (від gap-критика): якість cron/нагадувань, життєвий цикл фонових процесів (зомбі/orphan/втрата при рестарті), gate підтвердження деструктивних дій, маскування секретів у виводі, прогрес-егрес у Telegram, спостережуваність. **І немає eval-харнеса — будувати його першим.**

## 2. Критика поточного стану

### Файлові інструменти (`filesystem.py`)
- **`file_read`**: без cap/offset/limit/номерів рядків (`:40-50, 71-72`). Ціна: токени — щоб глянути рядок 4000 з 10K, поглинається весь файл (~80-120K токенів); файл 1 МБ перевищує вікно й валить хід; відсутність номерів рядків живить цикли «old_text not found». Бінарні/не-UTF-8 падають із непрозорим `Error reading file` (`:75-76`).
- **`edit_file`**: single-occurrence (`:198-202`), exact-match без fuzzy (`:194-195`), тихий no-op при `count>1`, немає diff/верифікації (`:205`). Ціна: rename N разів = N round-trips; кожен whitespace-промах → повний re-read файла + retry.
- **`list_dir`**: один рівень `iterdir()`, emoji-імена без шляху/size/mtime/ліміту (`:248-250`). Ціна: O(к-ть тек) викликів на орієнтацію; модель домислює шляхи (помилки, emoji в аргументі).
- **`write_file`**: мовчки перезаписує без бекапу/append (`:139-144`) — реальний ризик втрати нотаток.

### Пошук / grep / glob
- Інструмента пошуку немає ніде; єдиний шлях — `exec+grep` із трьома класами відмов: подвійне екранування (`shell.py:73`), head-усічення 10K (`shell.py:116-118`), false-positive safety-guard на `/`/`../` (`shell.py:141,147-155`) — саме коли воркспейс залочено. Немає glob, немає видобування PDF (тільки 5 форматів зображень, решта → UnicodeDecodeError).

### Shell exec і фонові процеси
- **Усічення head-only 10K** ховає stderr+код виходу в кінці → зламана команда читається як успішна (`shell.py:100-118`). **Timeout** повертає лише рядок помилки, викидаючи весь буфер (`:96-98`). Таймаут фіксований 60 с без параметра (`:17,49-63`). **Orphan grandchildren**: kill б'є лише `/bin/sh`, без `start_new_session`/killpg і без reap на timeout-гілці → зомбі накопичуються. Немає `stdin=DEVNULL` → інтерактивні команди висять увесь таймаут.
- **Фонові**: задачі лише в пам'яті (`background.py:121`) — губляться при `restart`/`update`; незахищений `process.wait()` на демонізації навічно тримає 1 з 10 слотів (`:188-189`); буфер — дека на 1000 *рядків*, не байтів, без per-line cap (`:48-49`) → один minified-рядок ≈ 12.5K токенів.

### Веб-інструменти (`web.py`)
- `web_fetch`: head-only усічення без offset (`:160,196-198`); звітує **пост-усічену** довжину → агент не знає, втрачено 1% чи 99% (`:196-201`); немає кешу/дедуплікації; regex HTML→markdown ламає таблиці/списки/код (`:205-215`); відносні лінки не резолвляться (`:208-209`); **SSRF** — `_validate_url` пускає `localhost`/`127.0.0.1`/`169.254.169.254` при `follow_redirects=True` (`:34-44,175-176`); тіло цілком буферизується в пам'ять; великий JSON усікається в **невалідний** JSON (`:185-186`); немає retry на 429/5xx. `web_search` віддає плаский рядок, не структуру, без пагінації >10.

### База / контекст / цикл
- Немає центрального бюджету виводу (`registry.py:60`, `context.py:440-446`). Немає реального ліміту ітерацій — `while True` (док бреше про 20). Flush керується wall-clock TTL кеша (`cache.py:75-88`) → недетермінований: та сама розмова, програна повільніше (>5 хв), мовчки вирізає результат, який агент ще читає. Немає task/todo-інструмента, помилки stringly-typed (модель не може розумно ретраїти), немає дедуплікації повторних читань.

### Сліпі зони, не охоплені ідеацією (gap-критик)
- **Cron/нагадування** (`cron.py`) — флагманська функція асистента, але нуль ідей: немає парсингу natural-language часу, прев'ю «next fire», коректності таймзон, list/edit/validate існуючих задач.
- **Життєвий цикл фонових процесів** — конкретні діючі дефекти (orphan/zombie/slot-leak/втрата при рестарті), жодна ідея не адресує, крім усічення виводу (#25).
- **Gate деструктивних дій + audit log** — shell+file+web від LLM-аргументів без human-in-the-loop; один галюцинований `rm`/порожній `write_file` незворотний.
- **Маскування секретів у виводі** — ніщо не вичищає `.env`/ключі, коли `file_read`/`exec` тягне їх у контекст і JSONL.
- **Прогрес/typing-егрес у Telegram** — агент може мовчати багато ітерацій; немає liveness-сигналу.
- **Спостережуваність** (per-tool latency/token/error-спани), **time/timezone-інструмент** для відносних дат.

## 4. Експериментальні ідеї (validate-before-commit)

Кожна — гіпотеза + ефект + spike-план. Будувати **тільки** після проходження gate'у.

- **#32 MCP-адаптер (L).** *Гіпотеза:* один config-driven шар (`mcpServers`) розблоковує календар/пошту/контакти, яких бот сьогодні не вміє взагалі. *Ефект:* розблокування можливостей, не токенів; цифри «77k→8.7k / 49%→74%» матеріалізуються лише при багатьох інструментах — на 20 поточних незастосовні. *Spike:* підключити 1 малий сервер (напр. календар) з lazy-завантаженням схем (meta-pair `mcp_search`/`mcp_call`), allowlist + явний opt-in per server; виміряти tool-error-rate MCP-викликів і токени схем eager-vs-lazy. *Ризик:* зовнішні сервери з привілеями бота, який вже має shell — реальна експозиція.
- **#27 Семантичний шар на `search_memory` (M).** *Гіпотеза:* embeddings ловлять парафраз/синоніми, які FTS5 пропускає («ремонтую Volvo» vs «запис на ремонт авто»). *Ефект-ціль:* +10-25pp recall@5, але при single-user масштабі FTS5 може вже бути достатнім — ризик embedding cargo-cult. *Spike:* поверх #26; 20-задачний recall-сет із **парафразованими** запитами; **залишати лише якщо ≥ +10pp** над keyword; трекати latency/cost embedding-виклику/хід.
- **#31 `batch`-інструмент (M).** *Гіпотеза:* батчинг незалежних викликів стискає K round-trips→1. *Ефект:* виграш латентності/round-trip на fan-out; «37%» — це повний code-mode (не цей інструмент), «−60-80%» завищено (контекст усе одно росте). *Spike:* спершу інструментувати частоту незалежних fan-out'ів у логах; будувати лише якщо часті; чіткі per-call failure-семантики; не дати моделі батчити залежні виклики.
- **#17 `multi_edit` (M).** *Гіпотеза:* атомарні багатоблокові правки захищають неверсіоновані файли. *Ефект:* атомарність — головна (вузька) цінність; «105→21 викликів» — vendor self-report з кодового навантаження, не переноситься; реальна застосовність MEDIUM. *Spike:* виміряти частоту багатоблокових правок; інжект mid-batch fail → assert байт-ідентичний rollback.
- **#29 Структурна дистиляція фактів у JSONL при flush (M).** *Гіпотеза:* структуровані `{date,type,text}` шукаються точніше за прозу. *Ефект:* recall@5 ↑ vs проза; «11x» — запозичено. *Spike:* потребує #26/#28; порівняти recall на `facts.jsonl` vs сира проза; запінити схему промпта проти дрейфу.
- **#21 Disk-spill + pointer для надвеликих результатів (M).** *Гіпотеза:* запис повного payload'у у файл + вказівник зберігає інфо й уникає повторного запуску неідемпотентних команд (deploy/міграція). *Ефект:* 100% збереження, −1-3 re-run. *Spike:* залежить від #6 (offset/limit); потрібен cleanup (50 файлів/50MB); поважати profile root.

**Низький ROI / здебільшого надлишкові — валідувати потребу, не будувати наосліп:**
- **#24 JMESPath-проекція web_fetch (4.0)** — Telegram-асистент рідко б'є сирі JSON-API; нова залежність + модель має дати валідний JMESPath. Будувати лише якщо логи покажуть часті JSON-API виклики.
- **#20 Token-aware кепи замість char (4.7)** — другорядний полиш над #19; `estimate_tokens` сам ≈4-симв. наближення.
- **#12 Session-level дедуплікація читань (4.7)** — ≈0 для single-turn чату; цінність лише для довгих cron/sub-agent циклів; за прапорцем.
- **#11 `read_files` batch (4.0)** — поглинається #31; токен-виграш ≈0 (той самий контент). Лише якщо тривіально дешево.
- **#10 `stat`-інструмент (5.0)** — надлишковий, якщо #6/#7 приземляться; окремо не будувати.

## 5. Інструменти, яких бракує і варто додати

**Нові інструменти (з майстер-списку):**
- `grep(pattern, path='.', output_mode, glob=None, case_insensitive=False, context_lines=0, max_matches=50, offset=0)` — пошук по вмісту; rg + Python-fallback; у ОБОХ реєстрах.
- `glob(pattern='**/*.md', path='.', sort='mtime', limit=100, modified_within=None, fuzzy=True)` — виявлення файлів зі шляхами+метаданими.
- `search_memory(query, scope='all', top_k=5)` — ранжований FTS5/BM25 пошук по нотатках/MEMORY.md.
- `recall_session(query, after=None, before=None, limit=5)` — крос-сесійний пошук історії (строго per user_key).
- `task(action, text='', id='', notes='')` — довговічний план, що інжектиться в промпт і переживає компакцію.
- `multi_edit(path, edits[], dry_run=False)` — атомарна серія правок.
- `batch(calls[])` — паралельний fan-out існуючих інструментів (експ.).
- `stat(path)` / `read_files(paths[])` — лише якщо #6/#31 не приземляться.

**Нові можливості від gap-критика (вище за частину табличних ідей за пріоритетом):**
- **Час/таймзона:** `now(tz)` + `resolve_relative_date('next Tuesday', tz)` — детермінована арифметика дат для cron/нагадувань (stdlib `zoneinfo`, S). Прибирає off-by-one-day.
- **Хардненг cron:** розширити існуючий cron natural-language парсингом, прев'ю «next fires: Mon 2026-06-29 09:00 CEST», коректністю таймзон, list/edit/validate (M). Це флагман асистента — пріоритет високий.
- **Gate деструктивних дій + audit log:** confirm-крок для `rm`/`write_file`-clobber/`kill`/вихідних send'ів + журнал дій (M). Найвища severity зі сліпих зон.
- **Маскування секретів у виводі:** фільтр, що вичищає `.env`/ключі/токени з виводу `file_read`/`exec` до входу в контекст і JSONL (крос-cutting шар, M).
- **Запобіжник ітерацій/вартості:** реальний `max_iterations`, детекція осциляції двох інструментів, per-turn token/$ ліміт (S). Закриває необмежені витрати, особливо в cron/hook/heartbeat.
- **Прогрес/typing-егрес:** дати агенту typing-action і проміжні «крок 2/4» / редагування placeholder-повідомлення (M) — Telegram-специфічний UX.
- **Спостережуваність:** per-tool latency/token/error-class спани (LiteLLM уже віддає usage) — рантайм-половина до офлайн-eval (M).
- **Хардненг фонових процесів:** killpg + reap на всіх гілках, `stdin=DEVNULL`, wall-clock guard на `_run_job`, byte-bounded буфери, персистенс/reattach через `restart`/`update` (M).

## 6. Відкинуті ідеї та чому

**Явні «НЕ БУДУВАТИ» (підтверджено правильними):**
- **#5 ast-grep / структурний пошук** — вбито 2 лінзами (kill/kill/keep). Skeptic: IDE cargo-cult для markdown/нотаток/конфігів; додав би бінарну залежність + зайву схему в кожен промпт і **розмив би сигнал вибору grep-vs-ast**; ripgrep-grep покриває ~100% реалістичних запитів. Повертатись лише якщо з'явиться persona редагування справжнього дерева коду.
- **#18 apply_patch / unified-diff як ФОРМАТ ВВОДУ правок** — вбито 2 лінзами. Skeptic: імпортує відомий клас LLM-відмов (відсутні +/− префікси, фрагментовані hunks) **без вигоди** на файлах <500 рядків; Aider показує 2-15pp коливань syntax-fail. Потреби покрити дешевше: `multi_edit` + diff/`dry_run`. Unified-diff — лише як ВИВІД для читання, ніколи як ввід.

**Цифри, яким не вірити (флаги skeptic'а — не цитувати як обіцянку):**
«~90%/92%» grep (vendor/анекдот) · «−58/−47/−22%» glob (кодові репо) · «100-200x» file_read (крайній випадок) · «50%→90%» edit (вигадано) · «70%→90-95%» fuzzy-edit (Morph, інший домен) · «41-80% кеш» бюджет (інша інтервенція) · «0/20→16-18/20», «73%», «+20-30» search_memory (запозичено) · «+23-30» semantic (Mem0 multi-hop) · «+10-25pp» task (без джерела) · «37%» batch (це повний code-mode, не batch) · «40%→80%» eval (SWE-bench аналогія) · «105→21» multi_edit (vendor self-report). У всіх випадках **механізм/виправлення бага реальні — цифра не доведена**; eval-харнес (#33) існує саме щоб замінити їх виміряними.

## 8. План вимірювання

**Субстрат — golden-task replay (#33):** `tests/golden/` + `ragnarbot eval`, що ганяє ~30-50 курованих персональних задач через `AgentLoop.process_direct()` в ізольованому temp-воркспейсі, кожна з **детермінованим чекером** (файл існує/містить підрядок; нотатка досяжна через `search_memory`; cron заплановано; правка приземлилась; web-результат містить очікуваний токен).

**Метрики на задачу/хід:**
- **pass@1** і **pass^3** (детекція нестабільності).
- **tool_calls_per_task** і **LLM round-trips/task** (цикл уже рахує ітерації).
- **tokens/turn** і піковий/сумарний input-токен сесії (інструментувати токени-на-результат-інструмента).
- **tool-error-rate за класами** (not-found, ambiguous, timeout, shell-quoting, safety-guard-reject).
- **edit-success** (applied_correctly/attempts; з/без line_numbers; на whitespace-варіантах).
- **retrieval recall@5 / F1** на синтетичному 6-міс. корпусі нотаток (парафразовані запити) і на крос-сесійному сеті.

**Цільові субсети (по одному на спроможність):** пошук («знайди згадки X» — tool-calls vs exec+grep) · виявлення файлів («що змінилось сьогодні» — median 1 виклик, 0 reject'ів) · file_read-пагінація («відповідь з рядка N» — токени/read, к-ть «prompt too long») · правки (whitespace-варіанти — success + цикли re-read) · пам'ять (recall@5 vs grep; токени зони пам'яті до/після) · крос-сесія (к-ть досяжних відповідей до/після) · планування (next-fire коректність, off-by-day).

**Детермінізм і вартість:** cassette-style record/replay відповідей провайдера **або** дешева модель; ганяти **щоночі**, не блокуючи кожен PR. Валідація самого харнеса — навмисно зрегресувати інструмент і підтвердити, що golden червоніє.

**Рантайм-половина (gap):** per-tool latency/token/error-class спани (OTel/Langfuse; LiteLLM уже емітить usage) — щоб (а) перетворити заявлені дельти цього батчу на **виміряні продакшн-числа**, (б) знайти p99-вузьке місце й домінантний клас помилок, (в) ловити doc-vs-code drift у проді (як відсутній `max_iterations`).

**Принцип:** усі числа з розділів 3-4, позначені «*», — **baseline-then-delta гіпотези**. Жодну не публікувати як результат, доки відповідний golden не виміряє її на цьому воркспейсі.
