## Build & Development

```bash
# Install (dev mode)
pip install -e .

# Install with dev dependencies
pip install -e ".[dev]"

# Lint
ruff check ragnarbot/

# Run all tests
pytest tests/

# Run a single test file
pytest tests/test_tool_validation.py

# Run a specific test
pytest tests/test_tool_validation.py::test_name -v

# Verify config schema loads
python -c "from ragnarbot.config.schema import Config; Config()"
```

## Architecture

ragnarbot is an async-first, lightweight AI assistant (~4,700 LOC). The core loop:

```
Channel (Telegram) → MessageBus → AgentLoop → LLM (via LiteLLM) → Tools → Response → MessageBus → Channel
```

**MessageBus** (`bus/queue.py`) decouples channels from agent logic via async queues (inbound/outbound). This is the central communication backbone.

**AgentLoop** (`agent/loop.py`) is the processing engine: consumes inbound messages, builds context (system prompt + conversation history + skills), calls the LLM, and iterates on tool calls up to `max_iterations` (default 20). It also handles the `process_direct()` path for CLI and cron jobs that bypass channels.

**ContextBuilder** (`agent/context.py`) assembles the system prompt from workspace bootstrap files (AGENTS.md, SOUL.md, USER.md, TOOLS.md), memory, and skills. Skills use progressive loading: `always=true` skills are fully included in the prompt, others appear as summaries that the agent can load on-demand via `read_file`.

**Tools** inherit from `Tool` (abstract base in `agent/tools/base.py`) with `name`, `description`, `parameters` (JSON Schema), and `execute()`. Registered in `ToolRegistry` and exposed to the LLM in OpenAI function-calling format.

**Providers** abstract LLM access. `LLMProvider` base class → `LiteLLMProvider` implementation. Returns `LLMResponse` with content, tool calls, usage. Environment variables for API keys are set based on the model prefix in the default model string.

**Sessions** (`session/manager.py`) persist conversation history as JSONL files under the active profile root: `~/.ragnarbot/sessions/` for the default profile, or `~/.ragnarbot-<profile>/sessions/` for custom profiles. Keyed by `{channel}:{chat_id}`.

**Config** uses Pydantic models (`config/schema.py`), stored as JSON under the active profile root (`~/.ragnarbot/config.json` by default, or `~/.ragnarbot-<profile>/config.json` for custom profiles) with camelCase keys. The loader (`config/loader.py`) converts camelCase↔snake_case automatically. Supported providers: Anthropic, OpenAI, Gemini. Transcription (Groq Whisper) has its own `TranscriptionConfig` — it is a standalone tool, not an LLM provider.

## Git Workflow

**Every change** must be on a dedicated branch with a descriptive prefix:
- `feature/` — new functionality
- `fix/` — bug fixes
- `cleanup/` — refactoring, dead code removal
- `docs/` — documentation-only changes

Branch naming: `{prefix}/{short-kebab-description}` (e.g., `feature/add-slack-channel`, `fix/session-persistence-crash`).

Do not include AI/vendor markers in the meaningful part of the branch name. Avoid names such as `feature/codex-add-slack-channel`, `fix/claude-session-bug`, `cleanup/gpt-refactor`, or similar. If an external tool requires its own branch prefix, treat that as separate from the repository naming convention and keep the descriptive portion clean.

**Commits** follow [Conventional Commits](https://www.conventionalcommits.org/) format. One logical change per commit.

Format: `type(scope): short imperative description`

Types:
- `feat:` — new functionality
- `fix:` — bug fix
- `refactor:` — code change that neither fixes a bug nor adds a feature
- `docs:` — documentation only
- `test:` — adding or updating tests
- `chore:` — build, deps, config, CI

Scope is optional but encouraged when the change targets a specific module (e.g., `feat(tui): add provider selection screen`, `fix(session): prevent crash on empty history`).

For breaking changes, add `!` after type/scope: `feat(config)!: switch to TOML format`.

**Never include co-authorship attribution or AI-generated markers** in commit messages.

**Pull requests**: create via `gh pr create`. Title uses the same conventional format as commits (e.g., `feat: add Slack channel support`). Should be concise (<70 chars). Body: `## Summary` with bullet points describing what changed and why. **Never include "Generated by" attribution, test checklists, or test plan sections** in PR descriptions.

**Committing during implementation** is fine — commit freely as you complete logical steps of the plan. But once the main implementation is done and the developer starts manual testing, **do not commit bug fixes automatically**. When the developer reports a bug during testing, fix the code but **wait for confirmation that the fix works** before committing. Do not commit untested fixes — this avoids chains of broken "fix" commits.

**Push and PR** require explicit developer approval. Never push to remote or create a pull request without being asked. The developer will test manually first and say when it's ready.

When using plan mode, include the branch name in the plan.

## Key Conventions

- Python 3.11+, async/await throughout
- Pydantic for config validation, dataclasses for internal data structures
- Single CLI entry point via Typer (`cli/commands.py`)
- Config JSON uses camelCase keys externally, snake_case internally
- Skills are markdown files with YAML frontmatter in `skills/{name}/SKILL.md`
- All user data lives under the active profile root: `~/.ragnarbot/` for default, `~/.ragnarbot-<profile>/` for custom profiles
- `ruff` for linting (line-length 100, select E/F/I/N/W, ignore E501)
